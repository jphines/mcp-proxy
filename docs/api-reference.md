# API Reference

All endpoints are served on a single TLS 1.3 HTTPS listener. The default port is `8443`.

## MCP Endpoint

### `POST /mcp` — Send MCP message

The primary endpoint. Handles all MCP protocol messages (initialize, tools/list, tools/call, etc.) using the Streamable HTTP transport.

**Authentication**: `Authorization: Bearer <okta-jwt>`

**Content-Type**: `application/json`

**Request body**: MCP JSON-RPC message

**Response**: MCP JSON-RPC response (200) or SSE stream for streaming responses

**Session lifecycle**: The first POST with no `Mcp-Session-Id` header creates a session. Subsequent requests include `Mcp-Session-Id: <session-id>`.

**On unauthenticated request**: `400 Bad Request` (per MCP spec; `GetServer` returns `nil` which the SDK maps to 400).

**Tool name format**: Downstream tool names are namespaced: `{serverID}__{toolName}`. Example: `github__create_pull_request`, `clinical__get_patient_labs`.

---

### `GET /mcp` — Establish SSE stream

Opens a persistent Server-Sent Events connection for receiving server-initiated messages (tool list changes, notifications). Optional — clients that only need request/response communication can skip this.

**Authentication**: `Authorization: Bearer <okta-jwt>`

---

## OAuth Enrollment

### `GET /oauth/enroll/{serviceID}`

Initiates the OAuth 2.0 PKCE enrollment flow for the given service.

**Authentication**: `Authorization: Bearer <okta-jwt>`

**Path parameters**:
- `serviceID` — matches a server registration ID with `strategy: oauth`

**Response**: `302 Found` → redirects to the OAuth provider's authorization URL

**Error responses**:
- `401 Unauthorized` — missing or invalid bearer token
- `500 Internal Server Error` — failed to initiate flow (e.g., unknown serviceID, provider configuration error)

---

### `GET /oauth/callback`

Receives the authorization code from the OAuth provider after user consent.

**Query parameters**:
- `code` — authorization code from provider
- `state` — HMAC-signed state parameter (generated by `/oauth/enroll`)

**Response**:
- `200 OK` — enrollment successful; HTML page with "You may close this window" message
- `400 Bad Request` — missing parameters, invalid/expired state, or token exchange failure

**Security**: The state parameter is verified using HMAC-SHA256 with a server-side secret. Requests with expired (>10 minutes) or tampered state parameters are rejected.

---

## Approval Callbacks

### `POST /approvals/slack/callback`

Receives interactive component payloads from Slack (button clicks on approval messages).

**Content-Type**: `application/x-www-form-urlencoded`

**Slack headers required**:
- `X-Slack-Signature: v0=<hmac-sha256-signature>`
- `X-Slack-Request-Timestamp: <unix-timestamp>`

**Signature verification**: HMAC-SHA256 over `"v0:" + timestamp + ":" + body` using the configured Slack signing secret. Requests older than 5 minutes are rejected (replay protection).

**Response**:
- `200 OK` — decision received (even if requestID is unknown — prevents Slack retries)
- `400 Bad Request` — invalid or unparseable payload
- `401 Unauthorized` — signature verification failed
- `405 Method Not Allowed` — non-POST request

**Action IDs**: Slack button action IDs follow the pattern `approve_<requestID>` / `reject_<requestID>`.

## Observability

### `GET /metrics`

Prometheus metrics endpoint. Returns `text/plain; version=0.0.4` format.

**Key metrics**:

| Metric | Type | Labels | Description |
|---|---|---|---|
| `mcp_proxy_tool_calls_total` | Counter | `server_id`, `tool_name`, `decision` | Total tool calls |
| `mcp_proxy_tool_call_duration_ms` | Histogram | `server_id`, `tool_name` | Total proxy latency (ms) |
| `mcp_proxy_downstream_duration_ms` | Histogram | `server_id` | Downstream call latency (ms) |
| `mcp_proxy_credential_resolution_duration_ms` | Histogram | `server_id`, `strategy` | Credential resolution latency |
| `mcp_proxy_approval_wait_duration_ms` | Histogram | `server_id` | HITL wait time |
| `mcp_proxy_circuit_breaker_state` | Gauge | `server_id` | 0=closed, 1=open, 2=half-open |
| `mcp_proxy_active_sessions` | Gauge | — | Active MCP sessions |
| `mcp_proxy_downstream_errors_total` | Counter | `server_id`, `tool_name`, `error_type` | Downstream errors |
| `mcp_proxy_enrollment_required_total` | Counter | `server_id` | Enrollment-required blocks |
| `mcp_proxy_policy_eval_errors_total` | Counter | `server_id`, `tool_name` | Fail-open policy eval errors |

---

### `GET /healthz`

Liveness probe. Returns `200 OK` with body `ok` if the process is alive.

---

### `GET /readyz`

Readiness probe. Returns `200 OK` with body `ready` if the proxy is ready to serve traffic. (Phase 1: always `200`. Phase 2: checks database connectivity.)

---

## Error Response Format

Tool-level errors (authentication failure, policy denial, enrollment required, etc.) are returned as MCP `CallToolResult` with `IsError: true`:

```json
{
  "content": [{"type": "text", "text": "denied by policy"}],
  "isError": true
}
```

These are **not** JSON-RPC errors — they are tool results that indicate the tool call failed. This matches MCP spec §3.3.4.

**Protocol-level errors** (e.g., malformed JSON-RPC) are returned as JSON-RPC error objects per the spec.
